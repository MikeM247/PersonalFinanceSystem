# FR-002 Financial Year and Period Management Architecture

## Skill Invocation
- Skill: architect
- Scope: FR-002 Financial Year and Period Management
- Requirements Validated: no
- Enforcement Files Read:
  - .codex/enforcement/architecture.md
  - .codex/enforcement/adr.md
- Key Risks Identified: unresolved overlap and edit rules, ambiguity around custom versus calendar-year behavior, and inconsistent downstream reporting if active-year and current-period resolution are not standardized.

## 1. Problem Framing

### Goals
- Establish a canonical financial year model that planning and reporting features can reuse.
- Store generated monthly periods for each financial year so monthly, annual, and year-to-date flows share the same time structure.
- Provide a clear active-year model while still allowing explicit selection of non-active years for historical planning and reporting.
- Keep annual and monthly period relationships stable for downstream budget, transaction, dashboard, and reporting features.
- Preserve compatibility with future multi-user support by keeping the model user-scoped rather than globally singleton.

### Constraints
- MVP is single-user first, but the design must remain compatible with future multi-user support.
- The product is desktop-first for MVP and must keep the current planning context obvious.
- Period structures must support efficient monthly, annual, and year-to-date queries.
- Business logic must remain out of the UI and API layers and live in shared core services.
- The current workspace only implements auth persistence in `packages/db`; `apps/web`, `apps/api`, and `packages/core` are target architecture components, not current implementation reality.
- The design should remain portable across the repository's current SQLite-oriented package setup and the product document's longer-term MySQL target.

### Assumptions
- A financial year is one contiguous 12-month range aligned to full calendar months, not arbitrary mid-month dates.
- Calendar-year alignment is optional; custom start months are supported as long as the year still spans twelve full months.
- Monthly periods are generated by the system from year boundaries and are not manually created or edited one by one.
- Transactions remain keyed by `transaction_date`; fiscal membership is derived from generated period ranges rather than stored directly on each transaction.
- The active financial year is the default planning and reporting context, but the user may explicitly open other years without changing the active default.
- Once dependent budget or transaction data exists for a year, boundary changes are restricted to preserve stable references.

### Requirements Completeness
The feature requirements are directionally strong but incomplete for implementation. Architecture can proceed only by stating explicit assumptions and by calling out the decisions that still need product confirmation.

### Missing or Ambiguous Requirements
- It is undefined whether overlapping financial years are allowed for the same user.
- It is undefined whether the first created year must become active automatically.
- It is undefined whether historical years may remain editable after downstream budgets or transactions exist.
- It is undefined whether a financial year can be deleted or archived once referenced by budgets, reports, or imported transactions.
- It is undefined how the dashboard and reports determine the "current month" when the active year uses a non-calendar fiscal start.
- It is undefined whether the system needs year lifecycle states such as closed or archived in MVP, or whether timestamps and active context are sufficient.

## 2. Architecture Overview

### System Context
- `packages/db` will add the finance-period schema, migrations, and repository implementations for financial years and budget periods.
- `packages/core` will own year-boundary validation, period generation, overlap checks, active-year selection rules, and current-period resolution.
- `apps/api` will expose financial-year management and context endpoints under `/api/v1`.
- `apps/web` will provide setup and management screens plus the default year selector used by planning and reporting flows.
- FR-004, FR-005, FR-007, and FR-008 will consume this model rather than computing financial periods independently.

### High-Level Components
- Financial year management UI for create, rename, inspect, and activate flows.
- Financial year API surface for CRUD-lite operations and active-context resolution.
- Financial period domain service for validation, period generation, activation, and current-period lookup.
- Finance-period persistence layer for year, period, and user-context storage.
- Downstream context consumers in budget planning, reporting, and dashboard modules.

### Key Design Decisions
- Model financial years as explicit user-scoped records with start and end dates.
- Generate one annual `budget_periods` row and twelve monthly `budget_periods` rows transactionally whenever a financial year is created.
- Store active-year state as `users.active_financial_year_id` rather than `financial_years.is_active`.
- Keep transactions tied to natural dates and resolve fiscal membership through period date ranges.
- Treat monthly periods as system-managed records; boundary correction is allowed only before downstream dependencies exist, otherwise the user creates a replacement year.
- Create ADR-002 to record the canonical financial-period model because it is a cross-cutting decision for budgeting, reporting, dashboarding, and future imports.

## 3. Component Design

### Financial Year Management UI
Responsibilities:
- Render first-run setup when no financial year exists.
- Capture year name and boundary inputs with clear validation feedback.
- Display available years and indicate the active default context.
- Let the user inspect generated monthly periods before downstream budgeting begins.

Interactions:
- Calls `POST /api/v1/financial-years` to create a year and generate periods.
- Calls `GET /api/v1/financial-years` and `GET /api/v1/financial-context` for list and default-context views.
- Calls `POST /api/v1/financial-years/{id}/activate` to change the active default.

Failure Modes:
- Invalid boundary inputs are blocked with actionable validation messages.
- Missing active context after first-run setup leaves the UI in setup mode rather than guessing a year.
- API failure keeps the UI read-safe and avoids showing partial generated periods.

### Financial Year API Surface
Responsibilities:
- Validate request DTOs and translate them into core-service commands.
- Normalize financial-year and period responses for the web app and future clients.
- Enforce authenticated user ownership on every year and period request.
- Return stable error codes for overlap, invalid-boundary, not-found, and conflict cases.

Interactions:
- Delegates business rules to `packages/core`.
- Uses `packages/db` repositories for persistence and transactions.
- Exposes a shared context endpoint so downstream screens resolve the same default year and current period.

Failure Modes:
- Invalid payloads return `400`.
- Unsupported boundaries or malformed twelve-month ranges return `422`.
- Overlap or conflicting activation requests return `409`.
- Missing or foreign-user year identifiers return `404` or `403`.
- Persistence failures return `503` or a generic `500` with details only in logs.

### Financial Period Domain Service
Responsibilities:
- Validate that year boundaries span exactly twelve full months.
- Enforce user-scoped non-overlap and naming rules.
- Generate annual and monthly `budget_periods` records atomically.
- Resolve active financial year and current period for dashboard/report consumers.
- Block unsafe boundary edits once dependent records exist.

Interactions:
- Reads and writes through financial-year, period, and user repositories.
- Supplies context objects consumed by budget, reporting, and dashboard services.
- Reuses auth user identity from FR-001 for ownership and active-context storage.

Failure Modes:
- Incorrect boundary calculations can mis-bucket downstream actuals and budgets.
- Partial period generation would corrupt the time model, so creation must fail atomically.
- Activation of a year outside the authenticated user's ownership is rejected.
- Boundary changes after dependent data exists are rejected to prevent referential drift.

### Finance-Period Persistence Layer
Responsibilities:
- Persist `financial_years`, `budget_periods`, and the active-year reference on `users`.
- Expose indexed lookups for active-year resolution, per-user year listing, and period-by-date matching.
- Enforce foreign keys and uniqueness needed for stable downstream references.

Interactions:
- Core services execute year creation and period generation inside a single transaction.
- Downstream repositories consume period identifiers for budget lines and report queries.

Failure Modes:
- Missing indexes degrade report and dashboard performance as transaction history grows.
- Weak constraints allow duplicate or malformed period rows.
- Non-atomic writes can leave a year without its full monthly period set.

### Downstream Context Consumers
Responsibilities:
- Use the shared year and period model as the only planning and reporting time source.
- Resolve annual, monthly, and year-to-date views through selected year plus period identities.
- Avoid embedding fiscal date calculations in feature-local code.

Interactions:
- FR-004 annual budgets bind to `financial_year_id`.
- FR-005 monthly budget lines bind to `budget_period_id`.
- FR-007 and FR-008 derive current month and year-to-date windows from the shared context service.
- FR-006 transaction filters convert selected fiscal periods into date ranges rather than storing redundant period foreign keys.

Failure Modes:
- Feature-local fiscal calculations can diverge from the canonical period model.
- Dashboard and report modules can show different totals if they bypass the shared context resolver.
- Imports can misclassify data if they infer fiscal periods without using the generated ranges.

## 4. Data Model

### Entities

#### `users` update
- Add `active_financial_year_id` nullable UUID/TEXT foreign key to `financial_years.id`.
- Keep ownership anchored by the existing authenticated user model.

#### `financial_years`
- `id` UUID/TEXT primary key
- `user_id` UUID/TEXT foreign key to `users.id`
- `name` string, not null
- `start_date` date, not null
- `end_date` date, not null
- `created_at` timestamp
- `updated_at` timestamp

#### `budget_periods`
- `id` UUID/TEXT primary key
- `financial_year_id` UUID/TEXT foreign key to `financial_years.id`
- `period_type` enum `year | month`
- `sequence_number` integer where `0` represents the annual row and `1..12` represent fiscal month order
- `start_date` date, not null
- `end_date` date, not null
- `created_at` timestamp
- `updated_at` timestamp

### Relationships
- One `users` record has many `financial_years`.
- One `users` record references zero or one active financial year through `active_financial_year_id`.
- One `financial_years` record has exactly one annual `budget_periods` row and exactly twelve monthly `budget_periods` rows.
- Future `budgets` records reference `financial_years.id`.
- Future `budget_lines` records reference `budget_periods.id`.
- Transactions remain date-based and join to `budget_periods` by range for reporting and filtering.

### Integrity Constraints
- `financial_years.user_id` must be required so future multi-user support remains scoped cleanly.
- `financial_years.end_date` must be greater than `start_date`, and service validation must ensure the range spans exactly twelve full months aligned to month boundaries.
- Financial years must not overlap for the same user; this is enforced in the core service and should be backed by repository-level overlap checks.
- `budget_periods` must be unique on `(financial_year_id, period_type, sequence_number)`.
- Monthly `budget_periods.sequence_number` values must be contiguous from `1` through `12`; the annual row must use `0`.
- `budget_periods` date ranges must be contiguous, non-overlapping, and fully contained within the parent financial year.
- `users.active_financial_year_id`, when present, must reference a financial year owned by the same user; this ownership check is enforced by the service layer.

## 5. API Contracts

### `POST /api/v1/financial-years`
Purpose:
- Create a financial year and generate its annual and monthly periods atomically.

Input:
```json
{
  "name": "FY 2026",
  "startDate": "2026-01-01",
  "endDate": "2026-12-31",
  "activate": true
}
```

Output:
```json
{
  "financialYear": {
    "id": "uuid",
    "name": "FY 2026",
    "startDate": "2026-01-01",
    "endDate": "2026-12-31",
    "isActive": true
  },
  "periodSummary": {
    "annualPeriodId": "uuid",
    "monthlyPeriodCount": 12
  }
}
```

Error handling:
- `400` for invalid payload shape.
- `422` for invalid boundaries, non-month-aligned dates, or non-twelve-month spans.
- `409` for overlapping years or conflicting uniqueness rules.
- `503` if persistence is unavailable.

### `GET /api/v1/financial-years`
Purpose:
- List the user's available financial years with active-state projection and summary metadata.

Output:
```json
{
  "items": [
    {
      "id": "uuid",
      "name": "FY 2026",
      "startDate": "2026-01-01",
      "endDate": "2026-12-31",
      "isActive": true
    }
  ]
}
```

Error handling:
- `401` for unauthenticated requests.
- `503` for repository unavailability.

### `GET /api/v1/financial-years/{id}`
Purpose:
- Retrieve one financial year with its generated periods for setup, review, and downstream selectors.

Output:
```json
{
  "financialYear": {
    "id": "uuid",
    "name": "FY 2026",
    "startDate": "2026-01-01",
    "endDate": "2026-12-31",
    "isActive": true
  },
  "periods": [
    {
      "id": "uuid",
      "type": "year",
      "sequenceNumber": 0,
      "startDate": "2026-01-01",
      "endDate": "2026-12-31"
    },
    {
      "id": "uuid",
      "type": "month",
      "sequenceNumber": 1,
      "startDate": "2026-01-01",
      "endDate": "2026-01-31"
    }
  ]
}
```

Error handling:
- `404` when the year does not exist for the authenticated user.
- `503` for repository unavailability.

### `POST /api/v1/financial-years/{id}/activate`
Purpose:
- Change the authenticated user's default financial year context.

Output:
```json
{
  "activeFinancialYearId": "uuid"
}
```

Error handling:
- `404` when the year does not exist for the authenticated user.
- `409` when the year is not eligible for activation because of unresolved integrity issues.
- `503` for repository unavailability.

### `GET /api/v1/financial-context?asOf=2026-04-05`
Purpose:
- Resolve the default active financial year and the current fiscal period for dashboard and reporting bootstrap.

Output:
```json
{
  "activeFinancialYear": {
    "id": "uuid",
    "name": "FY 2026"
  },
  "currentPeriod": {
    "id": "uuid",
    "type": "month",
    "sequenceNumber": 4,
    "startDate": "2026-04-01",
    "endDate": "2026-04-30"
  }
}
```

Error handling:
- `404` when no active financial year has been configured.
- `422` when `asOf` falls outside the active year or is malformed.
- `503` for repository unavailability.

## 6. Non-Functional Requirements

### Performance Expectations
- Active-year and current-period resolution must be fast enough to run during app bootstrap and report/dashboard page loads.
- Year creation performs a small, fixed-size write set of one year row plus thirteen period/context updates and should remain effectively constant-time for normal single-user usage.
- Report queries that map transactions into fiscal periods must stay index-backed and avoid full-table scans across multiple years of history.

### Scalability Approach
- User-scoped year records preserve clean ownership boundaries for future multi-user support.
- Generated period rows grow at a predictable fixed rate of thirteen rows per financial year, which is operationally negligible.
- Shared context resolution centralizes fiscal-period logic so later reporting, imports, and dashboards do not copy or diverge from the same rules.

### Reliability Considerations
- Financial year creation and period generation must commit atomically or not at all.
- Active-year changes must update the user context in a single transaction-safe operation.
- If context resolution fails, the system must fail closed and require explicit setup rather than guessing a year or period.
- Boundary edits after dependent data exists must be blocked to preserve downstream referential integrity.

## 7. Observability

### Logging Strategy
- Emit structured logs for `financial_year.created`, `financial_year.updated`, `financial_year.activated`, `financial_year.activation_failed`, and `budget_periods.generated`.
- Include `request_id`, `user_id`, financial-year identifier, and validation outcome where relevant.
- Log rejection reasons for overlap and invalid-boundary attempts without leaking unrelated record contents.

### Metrics
- `financial_year_create_success_total`
- `financial_year_create_failure_total`
- `financial_year_activation_total`
- `financial_year_overlap_rejection_total`
- `financial_context_resolution_latency_ms`
- `budget_period_generation_failure_total`

### Health Checks
- Readiness checks must verify database connectivity because all year and period resolution depends on persistence.
- Period-generation consistency should be covered by integration tests and optional administrative diagnostics rather than heavyweight production health checks.
- Operational dashboards should surface spikes in create failures, overlap rejections, or context-resolution errors.

## 8. Execution Plan

### Epic 1: Finance Time Persistence Foundation

#### Story 1.1: Add financial-year and period schema support
- Introduce `financial_years`, `budget_periods`, and `users.active_financial_year_id` in the persistence layer.
- Acceptance criteria:
  - Schema supports user-scoped financial years and generated annual/monthly periods.
  - Foreign keys and uniqueness constraints prevent orphaned periods and duplicate sequence numbers.
  - Indexes exist for per-user year lookup, active-year lookup, and period-by-year access.

#### Story 1.2: Add transactional repositories for year and period creation
- Expose repository contracts and implementations that create a year and its generated periods atomically.
- Acceptance criteria:
  - Year creation cannot commit a parent year without its complete annual and monthly period set.
  - Repository APIs support overlap detection and active-year updates.
  - Repository tests cover rollback on partial-write failure.

### Epic 2: Core Financial Context Services

#### Story 2.1: Implement boundary validation and period generation
- Build framework-agnostic services that validate date ranges and generate contiguous fiscal periods.
- Acceptance criteria:
  - Only twelve-month, month-aligned boundaries are accepted.
  - Monthly periods are generated contiguously and cover the full parent year.
  - Overlapping years for the same user are rejected.

#### Story 2.2: Implement active-year and current-period resolution
- Add shared services for default context lookup and `asOf` date resolution.
- Acceptance criteria:
  - The service resolves the active year for the authenticated user.
  - The service returns the correct current fiscal month for custom or calendar-aligned years.
  - Missing active context is surfaced as an explicit setup condition.

### Epic 3: API and Web Setup Flows

#### Story 3.1: Expose financial-year management endpoints
- Add create, list, detail, and activate endpoints under `/api/v1`.
- Acceptance criteria:
  - API responses match the documented contracts and error semantics.
  - Authenticated ownership is enforced consistently.
  - Invalid boundaries and overlaps return deterministic, user-safe errors.

#### Story 3.2: Build financial-year setup and selection UI
- Add the first-run year setup flow and the active-year selector used by planning/reporting screens.
- Acceptance criteria:
  - A user with no year sees setup guidance instead of broken downstream screens.
  - A newly created first year can become the active default without additional manual wiring.
  - The active default is visible and changeable from the financial-year management surface.

### Epic 4: Downstream Integration and Hardening

#### Story 4.1: Integrate budgets, reports, and dashboard context consumption
- Wire shared year and period context into FR-004, FR-005, FR-007, and FR-008 foundations.
- Acceptance criteria:
  - Annual budget flows resolve the selected `financial_year_id`.
  - Monthly budget flows bind to canonical `budget_period_id` records.
  - Reporting and dashboard views use shared current-period resolution for year-to-date and monthly summaries.

#### Story 4.2: Validate time-model correctness end to end
- Add automated coverage for persistence, services, API contracts, and downstream context use.
- Acceptance criteria:
  - Unit tests cover boundary validation, period generation, and context resolution.
  - Integration tests cover atomic creation, overlap rejection, and activation updates.
  - End-to-end tests cover first-time setup, active-year switching, and cross-year dashboard/report resolution.

## 9. Risks / Trade-offs
- Generated period rows add persistence overhead, but they eliminate duplicated fiscal date logic across later features.
- Storing active-year state on `users` is simpler and more portable than enforcing a single active year row, but it places one finance-specific preference on the auth user record.
- Keeping transactions date-based avoids redundant foreign keys, but reporting depends on correct date-range joins and indexing discipline.
- Blocking boundary edits after downstream data exists preserves referential stability, but it introduces a replacement-year workflow when setup mistakes are discovered late.
- The current repository lacks `apps/web`, `apps/api`, and `packages/core`, so FR-002 architecture is implementation-ready conceptually but still depends on scaffold work beyond the existing persistence package.
